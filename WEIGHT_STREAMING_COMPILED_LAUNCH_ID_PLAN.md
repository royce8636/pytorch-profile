# Compiled Launch ID Plan For Exact Weight-Streaming Schedule Matching

## 1. Problem

The current weight-streaming plan is built from a runtime trace whose node IDs
and correlation IDs are generated by profiling. The current weight-streaming
hook in Inductor runs later, at wrapper-generation time, after fusion has
already changed the execution structure.

That creates a mismatch:

- the scheduler plan is expressed in terms of trace-level runtime nodes
- the Inductor hook sees final compiled wrapper launches
- many traced GPU nodes may collapse into fewer wrapper launches
- many CPU trace nodes do not survive as first-class wrapper insertion points

As a result, the current system uses snapping and rescaling rather than exact
matching.

If the goal is true 1:1 matching, the system needs a shared identifier that
exists across:

- compile/codegen
- runtime profiling
- llama bundle export
- schedule generation
- weight-streaming injection

## 2. Core Proposal

Introduce a new stable identifier:

- `compiled_launch_id`

This ID should represent one exact compiled wrapper launch site.

The main idea is:

1. assign `compiled_launch_id` during Inductor codegen, when the compiler still
   knows both:
   - the final compiled launch
   - the fused source nodes that produced it
2. emit that ID into runtime profiling output
3. export that ID into the llama bundle
4. build schedules keyed on `compiled_launch_id`, not trace node index
5. inject weight-streaming ops using that same ID during wrapper generation

This changes the matching problem from:

- "which runtime node probably corresponds to this wrapper line?"

to:

- "this profiled launch is exactly wrapper launch `L123`"

## 3. Why Existing Keys Are Not Enough

### Kernel names are not enough

Kernel names are useful labels, but not reliable IDs:

- the same kernel can launch multiple times
- fused kernels may have descriptive names, but that does not uniquely identify
  a specific launch site
- extern kernels and Triton kernels are mixed
- one needs launch-site identity, not only kernel identity

### Correlation IDs are not enough

Profiler-side IDs such as:

- `correlation_id`
- `linked_correlation_id`
- `rf_id`

are runtime/profiler identifiers. They are useful for reconstructing submit,
wait, and GPU relationships, but they are not compiler-stable launch IDs.

They do not answer:

- which exact wrapper launch site produced this event?
- what is the stable mapping back to the compiled wrapper?

### Trace node indices are not enough

The current plan keys off trace node positions. That fails once fusion reduces:

- many trace GPU nodes

into:

- fewer compiled wrapper launches

At that point, exact index-based alignment no longer exists.

## 4. Best Hook Point

The best hook point is not the raw FX graph and not the final post-hoc runtime
trace alone.

The best place is the codegen boundary where Inductor still knows:

- the final compiled launch site
- the fused `node_schedule` that produced it
- the final `kernel_name`

That is the point where launch identity is still exact and useful.

Conceptually, this is around:

- kernel definition / launch emission
- wrapper `KernelCallLine` creation
- extern launch line creation

At this point, Inductor can assign:

- one `compiled_launch_id` per emitted launch site

and record exact provenance before the information is lost.

## 5. Proposed End-To-End Flow

### Phase A: Compile-Time Launch Identification

During Inductor codegen:

- assign a monotonically increasing `compiled_launch_id`
- attach it to every emitted wrapper compute launch
- cover at least:
  - Triton kernel launches
  - extern kernel out launches
  - extern kernel alloc launches

For each launch, record metadata such as:

- `compiled_launch_id`
- launch ordinal within wrapper
- kernel name
- launch kind
- fused source node names
- provenance/debug handle if available

This metadata should be written to a compile-time sidecar file.

Suggested output:

- `compiled_launch_map.json`

## 6. Required Profiling Changes

Yes, profiling must change for exact matching.

The runtime/exported profile needs to expose the same `compiled_launch_id` that
was assigned during codegen. Without that, the scheduler still has no exact key
to target.

### Recommended approach

Wrap each emitted compiled launch with a marker carrying `compiled_launch_id`.

Possible mechanisms:

- `record_function("ws_launch:<id>")`
- NVTX range with `ws_launch:<id>`
- explicit lightweight profiler annotation if a lower-level mechanism exists

The important requirement is:

- the marker must be observable in the profiler trace
- the marker must be unambiguously associated with one compiled launch site

### Why markers are the cleanest approach

Markers let profiling discover the exact compiled launch site without relying on
heuristics like:

- kernel-name matching
- launch ordering guesses
- correlation-ID reconstruction

## 7. Llama Bundle Changes

The llama bundle should carry `compiled_launch_id` into the exported runtime
artifacts.

### `runtime_nodes.csv`

Add a new column:

- `compiled_launch_id`

Expected behavior:

- GPU runtime nodes corresponding to compiled launches should carry the ID
- related CPU submit nodes may optionally also carry the same ID
- unrelated CPU leaf nodes may leave it empty

### `ggml_profile_node_records.csv`

Also add:

- `compiled_launch_id`

This keeps the timing table aligned with the node metadata table.

### `runtime_edges.csv`

No structural change is strictly required, but the exported runtime graph can
benefit if edges connect nodes that already share a `compiled_launch_id`.

## 8. Scheduler Input Changes

The scheduler should stop building plans from raw trace node indices alone.

Instead, it should consume:

- `compiled_launch_id`

as the primary key for compiled GPU launch attachment.

That means schedule operations should evolve from:

- `before_node`
- `after_node`
- `evict_after_node`

toward fields like:

- `before_launch_id`
- `after_launch_id`
- `evict_after_launch_id`

Trace-node fields can remain temporarily for compatibility, but the exact path
should use launch IDs as the authoritative key.

## 9. Injection Changes In Inductor

Once schedules are keyed by `compiled_launch_id`, the wrapper hook no longer
needs to:

- classify trace nodes as GPU or CPU
- snap CPU nodes forward/backward
- rescale trace GPU indices to wrapper launch indices

Instead, the hook can:

1. walk final wrapper compute launches
2. read the launch's `compiled_launch_id`
3. attach scheduled prefetch/evict ops directly to that launch

This turns matching into a direct lookup rather than an approximation.

## 10. CPU Nodes And Synchronization

Exact GPU launch matching is the primary target.

CPU-side nodes are more complicated because many CPU trace activities do not map
to durable wrapper lines:

- submit calls
- waits
- sync calls
- staging/control ops

There are two options here.

### Option A: Keep exact matching only for compiled GPU launches

This is the simplest and most robust design.

Use `compiled_launch_id` for:

- scheduled prefetch attachment
- scheduled eviction attachment
- GPU-launch-level matching

Continue to reconstruct CPU nodes from profiling as supporting runtime detail.

### Option B: Add explicit wrapper-side CPU markers

If exact CPU-side schedule attachment is also required, the wrapper must emit
explicit markers for CPU-side phases such as:

- submit
- wait
- restore boundary
- evict boundary

Then the profiler can export those markers too.

This is possible, but it is a larger change and should likely be a second
phase.

## 11. Recommended Rollout

### Stage 1: Compile-time launch map

Add `compiled_launch_id` generation and sidecar export.

Deliverables:

- `compiled_launch_id` attached to wrapper launches
- `compiled_launch_map.json`

### Stage 2: Profiling visibility

Add profiler-visible markers carrying `compiled_launch_id`.

Deliverables:

- runtime events can be mapped exactly back to compiled launch sites

### Stage 3: Bundle propagation

Add `compiled_launch_id` columns to llama bundle CSV exports.

Deliverables:

- `runtime_nodes.csv` includes launch IDs
- `ggml_profile_node_records.csv` includes launch IDs

### Stage 4: Scheduler migration

Teach the scheduler and plan format to target launch IDs.

Deliverables:

- exact schedule attachment using `before_launch_id` / `after_launch_id`

### Stage 5: Direct injection

Update the weight-streaming injector to attach operations by
`compiled_launch_id`.

Deliverables:

- no snapping
- no fusion rescaling
- exact 1:1 launch matching

## 12. Benefits

This design provides:

- exact compiled-launch matching
- stable keys across compile and profile
- elimination of name-based heuristics
- elimination of index-rescaling heuristics
- compatibility with fused kernels
- better debugging because the schedule can be inspected in terms of actual
  compiled launches

## 13. Risks And Open Questions

### Stability across recompiles

`compiled_launch_id` should be stable for a given compilation artifact, but not
assumed to be stable across arbitrary recompiles if fusion decisions change.

This is acceptable if the intended workflow is:

1. compile
2. profile
3. build schedule
4. recompile with the same structural compilation settings

### Marker overhead

Profiler-visible launch markers add some overhead. The design should keep them:

- lightweight
- easy to disable outside schedule-building workflows

### Repeated launches of the same kernel

This is exactly why `compiled_launch_id` is needed. Repeated launches of the
same kernel name remain distinguishable by launch-site ID.

### CPU-side exactness

If exact CPU-side schedule attachment is required later, the design should be
extended with explicit CPU-side markers rather than inferred retrospectively.

## 14. Recommended Direction

The recommended direction is:

- keep the hook at the compiled-wrapper boundary
- add a compiler-assigned `compiled_launch_id`
- propagate that ID through profiling and llama-bundle export
- build the plan on that ID

This is the cleanest way to get true 1:1 matching without abandoning the value
of real post-fusion runtime traces.
